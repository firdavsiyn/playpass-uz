# PlayPass UZ — Full Audit (2026-06-18)

Multi-agent audit: 6 dimensions (payments, check-in fraud, RLS/authz, Flutter correctness, business model, perf/light-mode), every finding adversarially verified against real code. **35 filed → 30 confirmed → 5 refuted.**

> **Verdict: NOT launch-ready.** Both core loops (check-in, gift) are broken end-to-end, and several money/fraud holes are open. Most issues share one root cause: value-granting writes done client-side that must be SECURITY DEFINER RPCs.

## 🔴 Launch blockers (fix before any real user)

1. **Check-in fully broken — param contract mismatch.** Client sends `zone_id`/`qr_hmac`; the `checkin` function reads `club_id`/`qr_token` → every check-in returns HTTP 400. Core visit loop is dead. — `supabase_service.dart:485`. *Fix: send `club_id: zoneId`, `qr_token: qrHmac` (no server change).*
2. **clubs RLS leaks `qr_token` (check-in secret) + `payout_details` (bank cards) to every authenticated user.** `clubs_select_active` = bare `USING(status='active')`, `getClub()` selects all columns → anyone reads the QR secret and check-in remotely; club bank cards exposed. — `20250319000002_rls_policies.sql:59`. *Fix: move secrets to no-public-SELECT tables / revoke columns; explicit safe column list in getClub().*
3. **Server geofence bypassable.** `if (geo_lat && geo_lon && club.lat && club.lon)` — skipped entirely if request omits geo. With (2), zero physical-presence requirement → remote payout farming (8–18k UZS/visit). — `checkin/index.ts:154`. *Fix: when club has coords, require finite geo (403 if missing), always run haversine; add rotating per-scan token.*
4. **Gift flow: no money in, no entitlement out, no RLS.** Creation mints free certs with client-chosen plan/amount; redemption only flips `status='redeemed'`, grants no subscription; table absent from all 13 migrations; all checks client-side → self-issue/self-redeem/double-spend. — `supabase_service.dart:854-908`. *Fix: SECURITY DEFINER `redeem_gift_certificate(p_code)` atomic CAS + provision; gate creation behind payment webhook; add migration + RLS.*
5. **Gift screen sells banned UNLIMITED plans (bankruptcy vector).** `_buildForm` uses `AppConstants.plans.values` incl. legacy unlimited pro/vip/vip_annual; check-in treats them as infinite with no 1/day cap → up to 8×18k/day ≈ 4.3M payout vs 599k/2.5M price. — `gift_purchase_screen.dart:69`. *Fix: use `purchasablePlanCodes`; server CHECK rejecting non-purchasable on gift insert.*

## 🟠 High

1. **`activate_subscription_from_payment` RPC body is not in the repo** — the function that turns payment→subscription is unverifiable. — `20260610000001_visits_model.sql:49`. *Fix: commit it (daily→4, day→12+30d, anytime→12, never -1; idempotent on payment_id; DEFINER + locked search_path).*
2. **Day-Pass breaks the golden rule** — allows 4 peak visits → 72k payout vs 25k price. — `app_constants.dart:88`. *Fix: `daily` hours:1, slots:['day']; grant 1 off-peak visit.*
3. **Promo redemption records usage but never applies the bonus** (subscriptions UPDATE dropped by RLS). — `supabase_service.dart:570`. *Fix: apply hours/days inside `redeem_promo` RPC.*
4. **Click/Rahmat: activation failure after CAS→completed permanently loses the paid subscription.** — `click-webhook/index.ts:84-119`. *Fix: claim+provision in one transaction; re-activate on already-completed-but-missing; alert instead of silent console.error.*
5. **Password-reset deep link routes to `/home`, never reaches reset screen.** — `app_router.dart:95-114`. *Fix: special-case `type=recovery` → `/auth/reset-password`.*
6. **`redeem_promo` references non-existent columns** (`valid_until`, `usage_count`) → runtime error. — `20260506000002_security_hardening.sql:128`. *Fix: rename to `expires_at`, `used_count`.*

## 🟡 Medium
- Gift certs created without payment, never set to `paid` (unbacked revenue) — `supabase_service.dart:854`.
- `applyPromoCode` mutates subscriptions/promos client-side, bypassing the secure RPC — `supabase_service.dart:555`.
- Cooldown/daily-limit checks are non-atomic TOCTOU windows (burst evasion) — `checkin/index.ts:126`.
- Best-effort hours rollback writes a stale absolute value, clobbering concurrent decrements — `checkin/index.ts:214`.
- RLS-critical DEFINER fns (`is_superadmin`, `my_club_id`, `handle_new_user`) lack `SET search_path` — `20250319000002_rls_policies.sql:25`.
- Welcome bonus RPC `grant_welcome_bonus` is never called — new users never get it — `auth_screen.dart:138`.
- Global `favoritesProvider` keeps previous user's favorites after logout/login — `favorites_provider.dart:5`.
- `payout-calc` rates (10k/18k) match obsolete v1.2, not v1.3 (9k/14k/22-30k) — `payout-calc/index.ts:17`.
- BM's "1 visit/day" rule not enforced — only an 8/day ceiling — `checkin/index.ts:149`.

## ⚪ Low
- Gift redemption no atomic claim — same code redeemable by multiple users — `supabase_service.dart:903`.
- `loyalty_points` INSERT policy `WITH CHECK (true)` — any user forges ledger rows — `20260406000003_features_v2.sql:148`.
- Users can self-grant any achievement (client-trusted unlock) — `20260406000002_fix_rls_security.sql:81`.
- setState after async gap without mounted guard in scanner — `qr_scanner_screen.dart:135`.
- Active-subscription hero card still uses old-palette purple/indigo gradient (rebrand miss) — `my_subscription_screen.dart:313`.
- Main clubs list builds every card eagerly; each card watches the whole favorites set — `clubs_list_screen.dart`.
- Tariff codes/prices/caps match obsolete v1.2, not v1.3 — `app_constants.dart:99`.
- Dead/contradictory payout constants in checkin (`RATE_PER_VISIT_UZS=8000`, `clubRevenueShare=0.70`) — `checkin/index.ts:13`.

## Cross-cutting themes
1. **Client-side trust where DEFINER RPCs are required** — gift, promo, loyalty, achievements, XP all mutate sensitive rows from the client with broken/absent RLS. Move every value-granting write behind a DEFINER RPC; REVOKE direct table writes from `authenticated`.
2. **Money captured but entitlement never delivered** — gift/promo/welcome all grant nothing; activation RPC not in repo. Provisioning must be atomic with the money event and idempotent.
3. **Check-in presence/anti-fraud unenforced** — world-readable qr_token, opt-in geo, static daily token, TOCTOU caps. Needs rotating per-scan token + DB-enforced atomic limits.
4. **Code↔BM v1.3 drift** — gift sells banned unlimited, Day-Pass breaks golden rule, stale payout rates/tariffs, missing 1-visit/day cap. Land one coordinated BM-v1.3 migration.
5. **Non-atomic two-step writes** — webhook claim-then-activate, checkin check-then-decrement, promo read-modify-write. Wrap each in a single transaction/RPC.
