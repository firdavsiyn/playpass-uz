# PlayPass Audit Report

_Сгенерировано: 13 июня 2026 · 6 измерений, 74 находки, 28 подтверждено состязательной проверкой_


## 1. Health Verdict

PlayPass is a functional MVP with a **sound core architecture but two structural problems that dominate everything else**. The single biggest issue is **code↔BM-v1.3 drift**: the app still runs the old hours/unlimited pricing model (`day`/`anytime`, 149k/249k, unlimited VIP), while the documented business model mandates a visits-based plan structure (Lite/Plus/Student + one-time passes) with a fixed three-tier payout (9k/14k/22-30k). Because the database `subscription_plans` seed is the source of truth for charged price, and `payout-calc` independently encodes a stale 10k/18k two-tier rate, the *revenue-in* and *cash-out* sides are both wrong and mutually inconsistent — this is a money-correctness problem, not just cosmetics. The second cluster is **payment/check-in integrity**: a stranded-payment failure mode (money taken, no subscription), a missing-from-repo activation function, and a check-in flow whose only presence control (geo) is optional and whose QR token is a day-static, shareable secret — enabling payout farming. No findings indicate data-loss or unauthenticated account takeover, so this is **launch-blocking-on-money-paths but not a security fire**. Below those, there's a long tail of visits-vs-hours i18n mislabels and tech-debt that is low-risk and largely auto-fixable.

## 3. Medium — grouped by dimension

**BM-v1.3 model alignment (the dominant theme)**
- `subscription_plans` migration seeds wrong codes/prices (`day`149k/`anytime`249k); DB is the charged-price source of truth — must be superseded before v1.3 plans go live (`visits_model.sql:39`).
- Plan codes don't match BM (`daily`/`day`/`anytime` vs Lite/Plus/Student); no mapping layer (`app_constants.dart:82`).
- Off-peak time gate hardcodes `plan=='day'` and 08-18; BM is weekday 10-17/18 — wrong start hour, no weekday restriction, and goes no-op after rename (`checkin/index.ts:107`).
- `'daily'` pass = 4 visits/1 day instead of single trial entry — 4×9k payout vs 25k price violates golden rule (`app_constants.dart`).
- `simulate-payment` encodes pre-v1.2 hour-based plans incl. Unlimited (`simulate-payment/index.ts`).
- Visits-unit migration incomplete: `levelFromHours` thresholds, `referralBonusHours`, `hours_spent` still in hours (`app_constants.dart`).
- `checkin` tier/zone rules reference legacy plan names (`start`/`basic`); will misbehave after rename (`checkin/index.ts:121`).

**Security**
- Rahmat webhook signature scheme is unverified guesswork + non-constant-time compare; this endpoint activates paid subs (`rahmat-webhook/index.ts`).
- Wildcard CORS (`*`) on credentialed/payment functions (`click-create-payment` et al.).
- Internal error details (`payErr.message`, `String(e)`) leaked to clients across payment/QR functions.

**Bugs**
- Night/weekend visit detection uses UTC hours/weekday, not local — wrong classification in UTC+5 (`achievement_service.dart`).
- Achievement night/weekend/unique-club counts only see last 20 visits — 3 achievements unreachable for power users (`achievement_service.dart:25`).
- Snackbar shown on popped dialog context after `Navigator.pop` (`add_review_dialog.dart`).
- `playerStatsProvider` force-unwraps `currentUser!` — crashes when unauthenticated (`player_stats_screen.dart`).
- Active sub shows "expired" on its final valid day (date-vs-datetime mismatch) — user loses scan access on a paid day (`subscription.dart`).
- Freeze toggle: non-atomic read-modify-write of `end_date`, no double-tap guard — drift on rapid taps (`supabase_service.dart`).

**UX / i18n (visits-vs-hours copy drift)**
- Plan card shows `ч/мес` / `soat/oy` for a visits count (`my_subscription_screen.dart:793`).
- Gift purchase shows `ч`/`soat` for visits (`gift_purchase_screen.dart:110`).
- Referral share text promises `+3 часа`/`+3 soat` (`strings_ru.dart:462`).
- FAQ `faq.a6`, streak `bonus_3h`, player-stats `Часов` label all still say hours.
- Loyalty privileges hardcoded RU + reference removed hours rewards; renders RU in UZ locale (`loyalty_screen.dart:288`).
- Onboarding slides hardcoded RU, advertise non-purchasable legacy tiers + stale 149k price (`onboarding_screen.dart:57`). Note: app is RU-only by design; fix the stale copy, not "add l10n."
- Happy Hours screen hardcoded RU; raw exception strings shown as error UI across 8 screens.

**Performance**
- All-clubs list fetched 3-4× via separate providers (list/map/nearby/home) (`clubs_list_screen.dart:50`).
- Club thumbnails decoded at native resolution into tiny slots — no `memCacheWidth` anywhere but `app_avatar.dart` (`clubs_list_screen.dart:1055`).
- Club lists use `ListView(children:[...map])` not `.builder` — eager widget construction + `.map().toList()` on every rebuild (`clubs_list_screen.dart:866`).
- `getNearbyClubs` pulls full table, haversines client-side — should be a Postgres RPC (`supabase_service.dart`).
- `_FullPhotoViewer` builds + leaks a `PageController` in `StatelessWidget.build()` (`club_detail_screen.dart:707`).
- Per-card occupancy `Consumer` rebuilds all cards on any map mutation — needs `.select` (`clubs_list_screen.dart:1167`).
- Bottom-nav `BackdropFilter` blur rebuilt on every tab switch + locale read (`app_router.dart:298`).
- `use_build_context_synchronously`: 3 real async-gap risks, not just style (`friends_screen.dart:171`, `auth_screen.dart:91`, `add_review_dialog.dart:91`).

**Tech-debt (dead code / drift, medium)**
- `clubRevenueShare=0.70` — dead AND contradicts BM (revenue-share abolished); price-formatter logic copy-pasted in 5 places; check-in business-rule constants unused client-side; `_planGlowColor` only matches dead legacy codes; manual-payment placeholder constants (`XXXX`) still rendered next to the real online flow.

## 4. Low / Tech-debt (short list)

- **`dart fix --apply`** clears ~93 of 100 info lints (69 `prefer_const_constructors` + 15 curly-braces) in one shot.
- Deprecation time-bombs: Sentry v9 `autoAppStart`/`setExtra`, `Switch.activeColor` — migrate before next SDK bump.
- `RATE_PER_VISIT_UZS=8000` in `checkin` is dead/inconsistent — delete (payout-calc is source of truth).
- `loyalty_points` INSERT policy `WITH CHECK(true)` — lets any user inject spoofed history rows (no balance impact; balance lives on write-locked `users.xp`). Tighten to `user_id = auth.uid()`.
- `manual subscription_requests` trust client `amount_uzs` — gated by human admin who approves by plan code, so low real impact; resolve price server-side anyway.
- Dead code to delete: legacy `freezeSubscription`/`unfreezeSubscription`, `levelFromHours`/non-localized `levelLabel`, deprecated Yandex polling shims, non-autoDispose per-club detail providers (unbounded memory growth).
- QR-scanner `setState` after await without `mounted` guard (rare trigger) (`qr_scanner_screen.dart:178`).
- `ActiveSessionWidget` rebuilds full card every 1s — isolate the timer Text.
- Minor: TextEditingControllers in dialogs never disposed; booking date highlight ignores year; booking cards read a `photos` field the query never selects; friend accept/decline leaves `_busy` stuck on error; `localizedHoursSubtext` doesn't pluralize; `daily_secrets` auto-create race in `user-qr-generate`.
- Stale "will remove / temporary / backward-compat" comments — delete once the dead code above is removed.

## 5. Top 5 Prioritized Next Actions

1. **Fix the payment integrity path first (highest blast radius).** Commit the `activate_subscription_from_payment` body to a migration with the correct visits mapping, make activation idempotent, run activate-before-complete (or same transaction), and add a reconciliation sweep. This is real money-taken-no-value risk today.
2. **Land the BM-v1.3 model as one coordinated change (the biggest item overall).** Single migration superseding `subscription_plans` (Lite 99k/8, Plus 229k/12, + one-time passes), corrected `payout-calc` rates (9k/14k/22-30k with VIP/night tier), corrected off-peak gate (10:00 start, Mon-Fri, per-tier end), and matching Flutter constants/allowlists. Pricing-in and payout-out are both wrong and inconsistent — fixing one without the other breaks margins.
3. **Close the check-in fraud vectors.** Make geo mandatory and enforced unconditionally, replace the day-static club token with a short-lived/per-user proof, and remove the infinite-visit bypass for legacy unlimited subs (decoupling it from VIP-zone access). These three together stop payout farming.
4. **Stop selling banned/invalid products.** Gate `my_subscription_screen` and `gift_purchase_screen` to `purchasablePlanCodes`, fix the gift default plan, and move gift create/redeem behind a service_role function with server-derived price + tightened RLS.
5. **Fix the notification timezone bug + sweep the visits-vs-hours copy.** Set `tz.setLocalLocation` once in `init()`, then batch the i18n relabels (plan cards, gift, referral share, FAQ, streak, loyalty perks) and run `dart fix --apply` to clear the lint baseline. Low-risk, high user-visible polish.

**Note:** The largest single item is the **code↔BM-v1.3 mismatch** (plan codes/prices and payouts). It is correctly rated High rather than Critical because v1.3 is a documented proposal and the current model still functions — but it touches both revenue and cost sides through DB-level sources of truth, so it should be treated as the primary launch-blocking workstream and sequenced as action #2.
